# כללי UI (Strict) + RTL + פלטת צבעים

## עקרונות UI קשיחים (אכיפה ב־Design System)
ה־UI של המערכת חייב להיות עקבי, predictable, ונגיש. אנחנו מתייחסים ל־Design System כאל חוזה: רכיבים חדשים אינם נכנסים לפרודקשן אם הם לא בנויים על רכיבי הבסיס (Button, Input, Dialog, Toast, DataTable) וה־tokens המרכזיים.

כללים תפעוליים:
- **אין “סטיילינג נקודתי”**: לא מוסיפים `style=` inline ולא כותבים CSS אקראי בקומפוננטה. כל שינוי עובר דרך tokens (spacing/typography/colors) או הרחבה מבוקרת של רכיב.
- **סקייל טיפוגרפיה מוגדר**: כותרות/טקסט/הערות משתמשים רק בסולם גדלים שננעל מראש (למשל 12/14/16/20/24/32). אין חריגים ללא החלטה תיעודית.
- **מצבים (states) אחידים**: לכל רכיב חייבים states ברורים: default, hover, active, focus-visible, disabled, loading, error. Focus חייב להיות גלוי ובקונטרסט מספיק.
- **ריווח אחיד**: מרווחים הם כפולות של 4px בלבד. חריגה מותרת רק לצרכי pixel-perfect ברכיבי אייקון/טוגל.

## RTL: הנחיות פריסה והתנהגות
המערכת היא RTL-first. משמעות הדבר היא שהפריסה והטקסט נבנים כך שברירת המחדל היא `dir="rtl"` וההתאמות ל־LTR (אנגלית) הן המקרה המשני.

כללי RTL מחייבים:
- **שימוש בלוגיים של CSS**: להשתמש ב־`margin-inline-start/end`, `padding-inline-*`, `inset-inline-*` ו־`text-align: start/end` במקום left/right. זה מונע “שבירה” במעבר שפה.
- **אייקונים כיווניים**: חצים, chevrons, “קדימה/אחורה” חייבים להתהפך בהתאם ל־dir. יש להעדיף אייקונים שמאפשרים mirror.
- **טבלאות ורשימות**: עמודת “פעולות” (Actions) ממוקמת בתחילת השורה ב־RTL (כלומר בצד ימין), אלא אם הוחלט אחרת למסך ספציפי.
- **אנימציות והחלקות**: slide-in, drawers, sidebars נפתחים מצד ימין ב־RTL. אין hardcode לצד פתיחה.

## פלטת צבעים (Design Tokens)
החלטנו על פלטה ניטרלית + צבעי סטטוס ברורים. המטרה היא קונטרסט גבוה, מראה נקי, ותמיכה מלאה ב־dark mode.

Tokens עיקריים:
- **Primary**: כחול עמוק עבור פעולות ראשיות (Create, Save) — נבדק לקונטרסט AA לפחות.
- **Neutral**: מדרג אפורים לרקעים, גבולות וטקסט משני. טקסט ראשי לעולם לא יהיה אפור בהיר.
- **Success/Warning/Danger**: ירוק/כתום/אדום לשגיאות וסטטוסים. משתמשים רק בגווני הסטטוס (לא "אדום מותאם" מקומי).
- **Focus Ring**: טבעת פוקוס בגוון Primary בהיר, עם שקיפות קבועה.

כללי שימוש בצבע:
- **לא לצבוע מידע בלבד**: סטטוס חייב לכלול גם אייקון/טקסט/label כדי שלא יהיה תלוי בצבע.
- **רקעים**: רקעי סטטוס (success/warn/error) הם tint עדין; הטקסט נשאר dark/light לפי theme.
- **דארק מוד**: אין “היפוך צבעים” ידני; משתמשים במיפוי tokens ל־dark.

## רכיבי נתונים (Board/Table) — כללי צפיפות
למערכת יש צפיפות גבוהה (הרבה משימות/שדות). לכן מוגדרים 3 מצבי density:
- **Comfortable**: ברירת מחדל (ריווח נדיב, מתאים למובייל/טאבלט).
- **Compact**: למשתמשי power (יותר שורות על המסך).
- **Dense**: רק למסכי ניהול/אדמין עם אזהרה (נגישות עשויה להיפגע אם לא בודקים).

כל מסך שמציג רשימות חייב:
- לשמור header דביק (sticky) בעת גלילה.
- לספק empty state ברור עם פעולה מוצעת.
- לשמור על skeleton loading אחיד כדי למנוע layout shift.
